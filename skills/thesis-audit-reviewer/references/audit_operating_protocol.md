# 审阅执行协议

## 核心定位

本 Skill 的目标是把学位论文审阅从“读完后写意见”改为“证据驱动、覆盖可追踪、结论可复核”的审阅工作流。正式报告只是最终交付物，审阅本体是对象账本、审阅矩阵、问题库、事实核验表和覆盖报告。

## 任务路由

| 用户场景 | 默认动作 | 禁止事项 |
|---|---|---|
| 只问流程、计划、Skill 设计 | 解释流程或完善 Skill | 不启动完整论文审阅 |
| 上传 PDF 要正式审阅 | 建工作区，解析，建账本，矩阵审阅，出报告 | 不直接凭印象写总结 |
| 上传 DOCX 要批注版 | 使用 Documents 能力写入批注，本 Skill 提供审阅规则 | 不只给外部报告 |
| 用户未提供审查规则 | 使用 `default_review_sources.md` 中的自检清单和问题汇总 | 不临时发明审查标准 |
| 要联网核验 | 将外部事实登记到事实账本，再逐项联网核验 | 不把未核验事实写成确定错误 |
| PDF 抽取差、公式表格异常 | 回看原图或页面渲染，必要时标为 blocked | 不把抽取失败当作论文错误 |
| 多篇论文批量审查 | 每篇独立工作区，逐篇通过单篇门禁，再汇总 | 不把多篇证据、页码、问题库混用 |

## 标准阶段

### 1. 输入登记

记录源文件路径、文件名、页数、文件大小、SHA256、解析方式、是否有自检清单/问题汇总/学校规范。若论文版本不唯一，先确认版本，不要混审。

默认审查依据是 `default_review_sources.md`。如果用户上传了新版自检清单或问题汇总，应在 source manifest 中记录新版文件路径，并说明替代了默认依据。

默认输出模式按 `input_output_policy.md` 执行：PDF 输入产出 PDF 审查报告，DOC/DOCX 输入产出批注版 DOCX。

### 2. 解析与页码化

默认场景是可第三方解析的普通论文，PDF 默认优先 MinerU VLM 或等效版面解析；同时保留 PDF 物理页口径。若用户声明材料保密、未发表、限制上传或不得外传，应改用本地解析或先确认。

解析文本只能作为证据线索，公式、表格、参考文献、编号和跨页内容必须在高风险处回到 PDF 原图复核。

### 3. 对象账本

先列出审阅分母，再判断问题。至少登记：

- 页、章节、段落。
- 图、表、公式、模型、变量、数据来源。
- 参考文献条目和正文引用。
- 题目关键词、核心概念定义。
- 外部事实声明、强结论、政策建议。

未进入对象账本的内容，不应在覆盖率中声称已经检查。

### 4. 审阅矩阵

把自检清单、问题汇总、学校规范和通用学术规范拆成检查项。每个检查项必须有状态：

- `pass`：已检查，未发现问题。
- `issue`：已检查，发现问题。
- `needs_factcheck`：需要外部核验后才能判断。
- `needs_author_source`：公开渠道无法核验，需要作者提供材料。
- `not_applicable`：该项不适用。
- `blocked`：因解析质量、原图不清、材料缺失等无法判断。

禁止状态留空。`pass` 也要写明检查范围，避免“未写问题”等同“未检查”。

### 5. 问题库

只有具备位置、原文、判断依据和修改建议的问题才能进入问题库。同类问题可以合并，但合并后必须保留逐处定位。问题库中的每条问题必须能映射到审阅矩阵行或对象账本项。

### 6. 外部事实核验

事实声明边读边登记，不在报告阶段凭记忆补。核验结果分为：

- `verified`：权威来源支持。
- `partly_verified`：部分支持，但口径、年份或范围不同。
- `contradicted`：权威来源相反。
- `unsupported`：未找到可靠公开来源。
- `needs_author_source`：可能来自内部数据，需作者提供来源。

当前性事实、公司数据、政策法规、标准认证、产品发布时间、行业规模、市场份额和用户明确要求核验的内容必须联网。

### 7. 报告生成

报告从问题库、事实核验表、参考文献审计和覆盖报告生成。报告语言必须是正式审核/批注语言，不写方法论说明。每条批注固定包含定位、原文摘录、批注意见、审核依据、修改要求、优先级和确定性。

### 8. 完成门禁

最终交付前执行 `references/completion_gate.md` 和 `scripts/validate_audit_report.py`。若存在 blocked 项，必须在报告“覆盖与剩余风险”中列出，不得暗示已经完整判断。

## 防遗漏机制

- 每篇论文独立工作区，独立编号，不混用上一篇论文的问题库。
- 先建对象账本，再写问题。
- 每轮阅读后更新矩阵或问题库，不把线索留在聊天记忆里。
- 每个图、表、公式、参考文献至少有一次覆盖记录。
- 所有外部事实先入账本，再核验。
- 最终报告不得出现未进入问题库的问题。
- 问题库不得出现没有位置和原文依据的问题。

## 不能迁移的 UCAS 署名审查逻辑

UCAS 署名审查是单一硬规则任务，论文审阅不是。论文审阅不能把研究问题、文献综述、创新性、政策建议等判断完全规则化。应迁移的是工作系统：证据结构、覆盖记录、失败兜底和交付门禁；不迁移“单规则自动判定”的窄任务逻辑。
