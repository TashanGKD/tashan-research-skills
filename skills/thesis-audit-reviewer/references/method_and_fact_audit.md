# 方法前提与可验证声明审计

本文件把个案中的 AR/CAR 复算、交易日检查、金额对照等问题抽象成普适机制。不要把某一种方法的检查项硬编码为全局规则；应先识别论文使用的方法，再触发该方法对应的审计义务。

## 一、四类可验证对象

正式审阅必须区分四类对象：

1. **外部事实**：公司数据、政策、产品、标准、行业规模、历史事件、财务数据。
2. **内部一致性事实**：同一数字、年份、章节号、表号、变量名在摘要、正文、表格、结论、参考文献中是否一致。
3. **方法前提**：某种方法成立所依赖的日期、样本、窗口、变量、单位、分组、口径、数据频率、样本量等。
4. **作者自算结果**：论文根据数据和模型计算出的核心结果，例如效率值、回归系数、异常收益、主成分得分、聚类结果、统计检验结果、访谈编码统计。

## 二、方法识别

在正式审阅前建立 `method_profile`。每个方法至少记录：

- 方法名称。
- 论文位置。
- 方法在论文中的作用：背景说明 / 辅助分析 / 核心证据 / 核心结论来源。
- 输入数据。
- 关键参数、窗口、样本和变量。
- 输出结果。
- 是否支撑摘要、结论、创新点或政策建议。
- 是否可以由公开数据、表内数据或论文披露方法进行最小复核。

常见方法触发器：

| 论文方法 | 普适审计义务 |
|---|---|
| 事件研究、时间序列、政策冲击 | 检查事件日期、窗口、估计期、有效交易日/有效观测日、基准模型、核心结果是否可复算 |
| 回归、DID、面板模型 | 检查样本期、处理组/对照组、变量定义、数据来源、稳健性声称、系数/显著性是否与表格一致 |
| DEA、Malmquist、效率评价 | 检查 DMU 与指标数量关系、投入产出指标口径、导向/规模报酬设定、分解解释、核心效率结果是否可复核 |
| PCA、因子分析、综合评价 | 检查原始指标矩阵、标准化、KMO/方差贡献率/载荷、权重计算、累计贡献率逻辑 |
| 聚类、分类、Copula、统计套利 | 检查样本选择、距离/相关性度量、参数、阈值、回测窗口、交易成本、结果可复现性 |
| 问卷、量表、实验 | 检查样本量、回收率、有效样本、信效度、实验组/对照组、测量时间点 |
| 访谈、扎根理论、案例研究 | 检查访谈人数、材料条数、编码过程、阶段划分依据、材料来源、审计轨迹和饱和度 |
| 文献计量、知识图谱 | 检查数据库、检索式、检索日期、去重规则、时间范围、软件参数 |

## 三、核心结果复核规则

若某个作者自算结果满足以下三项，应触发最小复核：

1. 支撑摘要、章节结论、最终结论、创新点、政策建议或关键图表。
2. 论文披露了足够的方法、公式、参数或原始表格，或公开数据可以取得。
3. 复核成本在当前任务中可接受。

复核层级：

- **Level 0 表内一致性**：表格数值、正文描述、摘要/结论数字是否一致。
- **Level 1 公式代入/简单复算**：用表内数据验证比率、均值、增长率、累计值、贡献率等。
- **Level 2 公开数据最小复算**：用公开数据和论文披露方法复算核心方向、数量级或关键结果。
- **Level 3 完整复现**：需要作者原始数据、代码、软件版本或内部资料；若缺失，标为 `needs_author_source`。

不能把所有可复核问题直接标为 `needs_author_source`。只有在 Level 0-2 均不可行时，才进入 Level 3。

## 四、可验证声明账本

所有可验证声明先进入 `verifiable_claims.csv`，再决定验证动作。至少包括：

- 数字、金额、百分比、样本量。
- 日期、年份、时期、窗口。
- 章节、图、表、公式、参考文献交叉引用。
- 数据来源、指标口径、估算方法。
- 强结论和政策建议中的具体数值或事实。

验证动作取值：

- `internal_crosscheck`：论文内部互相对照。
- `external_factcheck`：联网或查权威来源。
- `method_premise_check`：校验是否满足方法前提。
- `minimal_recalculation`：用表内或公开数据做最小复算。
- `author_source_required`：需要作者原始数据、代码、内部报告或源文件。
- `blocked`：解析、原图、数据或权限不足。

## 五、内部一致性审计

以下情况必须主动交叉验证：

- 同一对象的金额、年份、比例、样本量在不同章节或表格中重复出现。
- 正文说“见第 X 章/表 X/图 X/公式 X”，必须确认目标存在且编号正确。
- 摘要、英文摘要、正文、结论的研究对象、年份、方法名、缩写解释和核心数字应一致。
- 参考文献中的数据源年份、指数/数据库名称、URL 日期应覆盖正文使用的数据口径。

## 六、`pass` 判定标准

`pass` 只能用于完成了该对象所需验证动作的情况。若只是“看过结构”“粗读过一段”“登记了对象但未核验”，不得标为 `pass`，应使用：

- `needs_factcheck`
- `needs_internal_crosscheck`
- `needs_method_premise_check`
- `needs_recalculation`
- `needs_author_source`
- `blocked`

## 七、报告表达

不要写“作者应提供原始数据”来替代可公开复核的动作。应先说明已经完成或未能完成的复核层级：

- “该核心结果可由公开数据和论文披露方法进行最小复核，本次复核结果与论文数值不一致……”
- “该数值在正文不同位置出现两个版本，属于内部一致性问题……”
- “该方法依赖有效观测日/样本单位，但论文未说明该日期/样本是否满足方法前提……”
- “该结果需要作者提供原始数据和代码才能完整复现；在作者补充前，不宜作为确定性结论依据……”
